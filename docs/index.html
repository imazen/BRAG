The Biologically Rationalized Alpha-Grouped Pixel Format

Specification v1.0 — Ratified by the BRAG Standards Consortium — April 1, 2026

#![forbid(unsafe_code)] throughout. The entire stack. No exceptions.

Byte: [0] [1] [2] [3] B R A G ╰────┼────╭ The Compositing Triad™

Abstract

For decades, the pixel format community has accepted channel orderings designed around the limitations of display hardware circa 1987, convenient struct member alphabetization, and the historical accident of which engineer at Silicon Graphics ate lunch last.

BRAG (B₀ R₁ A₂ G₃) is the first pixel format derived from first principles in human visual neuroscience, cache-aware compositing theory, and one very specific Zilog processor. It is optimal. The argument is technically correct at every individual step while being collectively absurd.

This crate is the reference implementation. It also happens to contain the fastest u8 alpha compositor on crates.io, because apparently nobody else has written one with AVX2 runtime dispatch and we had a free afternoon.

Quick Start

use brag::{Bra, BRAG8, BRAG};

// The pixel type is Bra<G> — the signature spells BRAG.
// Green gets the generic because Green is all that matters.
let px: BRAG8 = Bra { b: 64, r: 255, a: 200, g: 128 };

// SIMD compositing (feature = "composite")
use brag::composite;
composite::premultiply(&mut pixels)?;
composite::src_over(&fg, &mut bg)?;

// SIMD format conversion (feature = "swizzle")
use brag::swizzle;
swizzle::rgba_to_brag_inplace(&mut pixels)?;

# Cargo.toml
[dependencies]
brag = { version = "0.1", features = ["composite", "swizzle"] }

§1 — Perceptual Justification

§1.1 — LMS Cone Fundamentals

ConePeak λDistributionRole
L ("Red")~564 nm63% of conesLuminance (dominant)
M ("Green")~534 nm31% of conesLuminance (secondary)
S ("Blue")~420 nm6% of conesChromatic only

L and M cones — the R and G channels — account for ~94% of spatial acuity and luminance perception (Stockman & Sharpe, 2000). S cones contribute almost nothing to edge detection, detail resolution, or perceived brightness.

§1.2 — Blue Spatial Acuity

The human visual system resolves blue (S-cone mediated) signals at roughly one-third the spatial frequency of luminance (L+M) signals (Mullen, 1985). Blue is, with scientific rigor, the least important channel.

Conclusion: B can go anywhere. We put it at byte 0, where it serves as a sacrificial prefetch preamble.

§2 — The Compositing Triad™

[B₀] [R₁ A₂ G₃] ↑ └──────────┘ meh The Compositing Triad™

One unaligned 32-bit read at byte 1 gets R, A, and G — all three operands for the critical compositing path. R×A and G×A are the perceptually critical products; errors in these terms are 3× more visible than B×A.

§3 — Historical Hardware Justification

§3.1 — The Zilog Z80 (1976)

; BRAG pixel from address HL:
LD BC, (HL)      ; B ← Blue,  C ← Red
LD DE, (HL+2)    ; D ← Alpha, E ← Green

; G×A needs E and D — SAME register pair. Zero cost.
; R×A needs C and D — ADJACENT pairs. One LD A,C away.

§4 — The Endianness Property

On little-endian (x86, ARM, RISC-V), a 32-bit load yields 0xGARB — which is what every other pixel format is compared to BRAG. This is not a coincidence. This is type theory.

On big-endian, the register contains 0xBRAG — which speaks for itself.

§5 — Conformance Requirements

  • MUST store Blue at byte offset 0
  • MUST store Red at byte offset 1
  • MUST store Alpha at byte offset 2
  • MUST store Green at byte offset 3
  • SHOULD use premultiplied alpha unless the user specifically requests otherwise, at which point the implementation SHOULD display a brief educational message about the superiority of premultiplied alpha before complying
  • SHOULD include at least one reference to the Z80 in its documentation
  • MAY refuse to convert to ARGB on philosophical grounds, provided a clear error message is displayed

§7 — FAQ

Is this a joke?

The crate compiles. The benchmarks are real. The vision science is real. The Z80 argument is real. Whether that makes it a joke is between you and your Cargo.toml.

Should I use BRAG in production?

Can you justify not using the perceptually optimal channel ordering? To your team? In the code review? We'll wait.

What does BRAG stand for?

Blue-Red-Alpha-Green. Or Biologically Rationalized Alpha-Grouped. Or Byte-Reordered for Architectural Gain. The acronym is flexible.

Why is Blue first?

Someone has to be. Blue drew the short straw perceptually (§1.2), so it draws the short straw positionally. Byte 0 is the foyer. Blue takes your coat.

What about GRAB?

Green at byte 0 violates blue-as-preamble (§1.2) and wastes prime real estate. Also, "grab" is already a verb with crate-namespace implications. Also, the endianness pun doesn't work.

I benchmarked BRAG against RGBA and they're the same speed.

On a Z80 they wouldn't be.

This was published on April 1st.

So was RFC 1149 (IP over Avian Carriers), which was later genuinely implemented with only 55% packet loss. BRAG achieves 0% packet loss. We are already more successful than carrier pigeons.
